In this exercise we will check & extend our application’s security, by verifying the encryption status of data in our system, and then by either:

  1. Adding an automated dependency scan to our CI pipeline
  2. Handling authentication of our users with OAuth

Part 1: Verify our application’s encryption

It’s important that any data our system has access to or control over is encrypted so that if (or when!) it falls into the wrong hands, they’re not able to access or use that data. This is of particular importance whenever the data itself might be sensitive, such as containing personal or financial details.

This is a good opportunity to review our application’s encryption.

Step 1: Verify our Encryption-at-rest

What is Encryption-at-rest?

Encryption-at-rest is a concept wherein persisted data (e.g. written to a hard drive or SSD) is encrypted. This can be done at various levels, but is often done transparently at the disk level so that the user of that drive isn’t even aware, but someone without the necessary passwords can’t decode that data. For example, your work laptop might be encrypted (with a tool such as BitLocker) so that if it were lost, the data on it wouldn’t be accessible without the relevant credentials.

Check it’s enforced

We’re not storing anything too sensitive right now, but we should understand whether we have encryption-at-rest for our database before we go any further. Take a look at the relevant Microsoft Docs for CosmosDB, and add a note to your Readme to discuss the status of your encryption.

Step 2: Verify our Encryption-in-transit

What is Encryption-in-tranit?

All the access control in the world for our application is of limited use if our responses (e.g. web pages full of data) are travelling over the internet in plain text, allowing attackers to potentially listen in to those messages. Encryption-in-transit is all about ensuring that all sensitive communication is only exchanged in an encrypted format between the parties involved. For internet communications, this is typically implemented with Transport Layer Security (TLS) which can be used to encrypt higher layer communications; for our application, we can leverage standard infrastructure by ensuring our application is only accessible over HTTPS (and not over basic HTTP) which use TLS to encrypt our HTTP communications and handles all traffic in an encrypted manner.

Check it’s enforced

Azure provides a domain name and handles certificates for encrypting traffic to that address automatically with an App Service, in fact you’ve probably already been accessing your application over HTTPS (you can see that the address in your browser starts with https:// not http://).

However, it’s important to acknowledge that having encryption available isn’t sufficient - we want to force everyone trying to use HTTP to switch over. Verify that’s the case now by both:

  1. Checking that the setting is turned on in the Azure Portal - you can find that under your app’s “Configuration -> General Settings” page.
  2. Testing it directly - try visiting http://<your_website> and check that it redirects you to HTTPS.

As a bonus - you should be able to see that the ssl=true query parameter is included in your CosmosDB connection string, ensuring that traffic between your application and the DB is also encrypted-in-transit by default.

Part 2: Extend our security

For this part of the exercise there is a choice of what you want to implement; choose one of the below options according to what would be most interesting or valuable to you. They are approximately listed in order from shortest to longest.

  • Dependency Checking: add a dependency checker to our CI pipeline to get regular feedback if security concerns are flagged in any of our libraries.

  • OAuth with Flask-Dance: implement authentication of users to our site using the OAuth code flow with the Flask Dance library
    • With GitHub
    • With GitLab
  • OAuth: implement authentication of users using the OAuth code flow, handled largely by you with the Flask-Login library to handle sessions - this is typically a more involved exercise
    • With GitHub
    • With GitLab

Submission Checklist

A non-exhaustive list of things to check before creating a Pull Request:

  • Your app forces the use of HTTPS on Azure
  • Your Readme discusses the current status of your encryption-at-rest
  • One of the subtasks has had the submission criteria met:
    • Dependency Checking
    • OAuth (GitHub/GitLab, with or without Flask Dance)
  • The app still satisfies the criteria from the previous exercises
    • Don’t forget to update the README and tests
